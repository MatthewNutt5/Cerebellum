    - Holds preferences like verbosity level

        be called something more general like data.json?
- implementations.json
    - Front-end
        - On startup, GUI reads implementations specified by the JSON, and provides
            a dropdown menu of these implementations
        - Each implementation spec starts with the name, e.g. CAENPowerSupply
            - This name will be used as the import name and display name
            - If an implementation for power supply, handled in a PowerSupply folder
        - Each spec also includes the options available and their type
            - E.g. IP is a string, baudrate is an int, etc.
            - Would be cool but even more complicated to specify dropdown options
        - So the base case of each power supply is displayName and implementation,
            then the implementation decides the config options available
    - Back-end
        - Then PSUConfig would need to be refactored; each implementation should
            include its own config, e.g. CAENPowerSupply has CAENConfig
            - These configs extend some PSUConfig interface, and EnvironmentConfig
                stores a list of these generalized PSUConfigs
        - Then the PowerSupply constructor calls the right implementation
            constructor depending on its config type
            - The constructor is given by the implementations.json
