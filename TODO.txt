Fixes/Refactors ==========

- Add more comments and update existing ones
- Move some imports into __init__.py to help simplify?
- The Great Renamening
    - EnvironmentConfig and EnvironmentControl are too similar in name
        - EnvironmentControl --> Controller
    - An EnvironmentConfig is always paired with a TestSettings
        - TestSettings --> TestConfig
    - Consider switching from camelCase to snake_case for variables and functions
- Generalize into Devices
    - Start with a Device interface, which only has init, del, and getID functions
        - Corresponding DeviceConfig only has displayName
    - PowerSupply will extend Device
        - PowerSupplyConfig will extend DeviceConfig
        - (Then each PowerSupply implementation can have its own Config)
    - ReadoutBoard, Oscilloscope(?) will extend Device
        - Ditto
    - Then, EnvironmentConfig holds a DeviceConfigList
        - Instead of indexing PSUs, events will index Devices
        - Will need a type check during execution
        - Add a pre-check function to TestSettings that takes a config and
            verifies that the events index into appropriate devices, e.g. a
            ReadoutBoard event won't try to index into a PowerSupply device
        - May want to refactor Events to split off into different Devices first
    - Events exist purely for one type of device
        - Some events, like WaitEvent, don't use any device
        - PSU events use a PowerSupply, etc.
    - PSUList becomes DeviceList, init function is called on all Devices

- implementations.json
    - Front-end
        - On startup, GUI reads implementations specified by the JSON, and provides
            a dropdown menu of these implementations
        - Each implementation spec starts with the name, e.g. CAENPowerSupply
            - This name will be used as the import name and display name
            - If an implementation for power supply, handled in a PowerSupply folder
        - Each spec also includes the options available and their type
            - E.g. IP is a string, baudrate is an int, etc.
            - Would be cool but even more complicated to specify dropdown options
        - So the base case of each power supply is displayName and implementation,
            then the implementation decides the config options available
    - Back-end
        - Then PSUConfig would need to be refactored; each implementation should
            include its own config, e.g. CAENPowerSupply has CAENConfig
            - These configs extend some PSUConfig interface, and EnvironmentConfig
                stores a list of these generalized PSUConfigs
        - Then the PowerSupply constructor calls the right implementation
            constructor depending on its config type
            - The constructor is given by the implementations.json
